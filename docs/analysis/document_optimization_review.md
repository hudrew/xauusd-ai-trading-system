# 文档优化评审

## 总结

当前文档体系已经足够支持 MVP 启动，但如果要进入“真实可开发、可回测、可运维”的阶段，仍有几类关键信息需要补强。

本次已经补入了高波动预警模块相关文档，但整体上仍建议继续优化以下内容。

## 建议优先优化的地方

### 1. 特征公式还不够精确

当前文档已经列出了大部分特征名，但很多字段还缺：
- 精确公式
- 计算窗口
- 单位定义
- 是否归一化
- 在回测和实时中的对齐方式

最需要补的字段：
- `breakout_distance`
- `pullback_depth`
- `range_position`
- `false_break_count`
- `realized_volatility`
- 高波动标签的定义方式

### 2. 标签定义还需要更严格

目前状态文档对 `trend_breakout`、`range_mean_reversion` 等定义已经不错，但还缺：
- 标签冲突时的优先级细则
- 状态切换的最短保持时间
- 状态抖动的去噪规则

高波动预警模块也一样，需要明确：
- 未来 5 / 15 / 30 分钟标签如何生成
- 什么叫“高波动”
- 预警触发后多久内不重复提醒

### 3. 接口契约还可以更细

技术架构文档已有模块划分，但建议继续补：
- 每个模块输入字段
- 每个模块输出字段
- 错误码
- 重试策略
- 时间戳规范
- 配置版本字段

### 4. 风控联动规则还不够具体

当前风控文档强调了原则，但还可以更细化：
- 高波动 warning 时是否只降仓位
- 高波动 critical 时是否直接禁开
- 区间策略在高波动状态下是否一律禁做
- 已有仓位在预警触发后如何处理

### 5. 运维文档还缺告警治理细节

建议继续补：
- 告警抑制时间
- 告警去重规则
- 多通道通知优先级
- 通知失败降级方案
- 日志保留策略
- 回放与审计归档策略

### 6. 回测验收标准还应补入预警维度

除了策略收益和回撤外，后续还建议增加：
- 预警命中率
- 误报率
- 平均提前量
- 重大事件前预警覆盖率
- 不同时段预警表现

## 当前项目主流程

建议把主流程理解为两条并行链路。

### 主链 1：交易决策链

1. 行情和事件数据进入 `DataIngestor`
2. `FeatureEngine` 计算并整理特征
3. `MarketStateClassifier` 输出市场状态
4. `StrategyEngine` 根据状态生成交易信号
5. `RiskManager` 做统一审核
6. `ExecutionEngine` 执行下单和状态同步
7. `ReportEngine` 输出复盘与报表

### 主链 2：高波动预警链

1. 行情和事件数据进入 `DataIngestor`
2. `FeatureEngine` 生成波动相关特征
3. `VolatilityMonitor` 输出未来时间窗的高波动风险评分
4. `AlertNotifier` 输出预警提醒
5. `RiskManager` 可按预警等级联动限制策略或仓位

## 重要节点

项目里最重要的不是“模型有多复杂”，而是以下节点是否定义清楚。

### 节点 1：数据进入系统

关键问题：
- 行情源是谁
- 事件源是谁
- 时间戳是否统一
- 缺失数据如何补

### 节点 2：特征生成

关键问题：
- 是否存在未来函数
- 多周期对齐是否正确
- 波动特征是否可实时更新

### 节点 3：状态识别

关键问题：
- 当前环境到底是趋势、回踩、区间还是不交易
- 状态切换是否过于抖动

### 节点 4：高波动预警

关键问题：
- 预警是否足够提前
- 原因码是否可解释
- 是否会频繁误报

### 节点 5：策略触发

关键问题：
- 环境和策略是否匹配
- 信号是否过密
- 信号是否可追溯

### 节点 6：风控审核

关键问题：
- 是否能真正阻断危险订单
- 高波动场景是否会自动降风险
- 风控是否能覆盖新闻、点差、滑点

### 节点 7：执行与同步

关键问题：
- 下单是否成功
- 订单状态是否和交易平台一致
- 异常后能否恢复

当前代码层面已经补到：

- `LiveTradingRunner`
- `MarketDataService`
- `ExecutionService`
- `MT5` 行情与执行适配器
- `cTrader` 统一接口与订阅请求构建

因此接下来最需要继续补的文档，不再是模块划分本身，而是：

- cTrader 异步会话连接时序
- 账户权益同步与持仓快照口径
- 订单状态回报字段定义
- 事件日历接入契约

### 节点 8：复盘与验收

关键问题：
- 回测是否包含真实成本
- 实盘与回测是否一致
- 预警模块是否真的提高系统稳定性

## 一句话建议

如果只看开发顺序，最应该优先补的是：

1. 特征公式和标签定义
2. 风控与预警联动规则
3. 接口契约和日志字段

只要这三块补扎实，后面的编码和迭代会顺很多。
